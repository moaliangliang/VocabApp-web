# VocabApp 优化方案

> **背景：** 基于竞品分析（墨墨背单词、百词斩、不背单词、红宝书）和用户反馈调研，对现有背单词 App 提出分阶段优化建议。

**目标：** 在不破坏现有架构的前提下，以最小开发成本提升核心学习体验

**当前版本功能概览：**
- 6 套词库（CET4/6、Kaoyan、TOEFL、IELTS、GRE），共 40000 词
- 浏览模式（展示词义音标例句，自动朗读，上下滑导航）
- 三阶段学习模式：选择题 → 翻卡 → 拼写
- SM-2 间隔重复算法
- 文章阅读（每课 15-20 篇）
- 看板（日历打卡、课程进度、里程碑成就）
- 单词详情页（词根词缀、同反义词、记忆技巧、快速标记）
- 设置（每日目标、补签）

## 二、竞品用户核心痛点

这些是竞品做得不好的地方，也是我们的机会：

| 痛点 | 涉及 App | 我们的机会 |
|------|----------|-----------|
| 图片依赖导致"假性记忆" | 百词斩 | 不依赖图片，直接用浏览模式展示真实信息 |
| 打卡焦虑/服务器依赖 | 百词斩 | 纯本地，无需联网，无断签焦虑 |
| 助记内容低俗 | 墨墨 | 词根词缀+算法生成，无人工擦边内容 |
| 旧词重复率过高 | 所有 App | SM-2 动态调整，mastery 3 自动降频 |
| 进度同步丢失 | 所有 App | 本地优先，无同步问题 |
| 词库不灵活 | 不背单词 | 已覆盖 6 种考试 40000 词 |
| 包体过大 | 红宝书 512MB | 仅 18MB |
| 必须买纸质书 | 红宝书 | 完全独立免费 |

## 三、分阶段优化提案

### Phase 1 — 快速见效（1-2 天/项）

#### 1.1 左右滑标记掌握状态

**在浏览模式增加左右滑手势，直接更新 mastery，跳转到下一个词**

```swift
// 左滑 → 不认识（quality: 0）
// 右滑 → 认识（quality: 4）
// 更新 LearningRecord + SM-2，自动进入下一个词
```

**工作量：** 修改 WordBrowseView + LearningViewModel 各约 20 行
**效果：** 用户可以在浏览模式快速过词，认识的直接跳过，不认识的进入学习流程

#### 1.2 展示词根词缀

**浏览模式展示 rootAffix 字段，WordDetailView 已有但浏览模式没有**

```swift
// WordBrowseView 加一行：
if !word.rootAffix.isEmpty {
    Text("词根: \(word.rootAffix)")
        .font(.footnote).foregroundColor(.secondary)
}
```

**工作量：** 修改 WordBrowseView 约 5 行
**效果：** 利用已有数据，帮助理解性记忆

#### 1.3 浏览模式显示掌握状态 + 下次复习时间

**根据 LearningRecord 展示 mastery 等级和复习倒计时**

```
掌握状态: [已掌握] 距离下次复习 3 天
           [学习中] 还需完成学习
           [未学习] 新词
```

**工作量：** WordBrowseView + 传参约 15 行
**效果：** 感知算法在起作用（墨墨核心卖点）

#### 1.4 错题熔断

**spelling 答错的词不进入下一个，而是重新进入 choice→card→spelling 兜底**

```swift
// advanceMode .spelling: 如果 !correct，不 advance，重新设置为 .choice（同一词）
```

**工作量：** LearningViewModel 约 5 行
**效果：** 烂词反复出现直到掌握

### Phase 2 — 体验提升（2-3 天/项）

#### 2.1 List 分组

**将 sessionWords 按 50 词一组分组，浏览模式显示 "第 3/20 组"+ 组内进度**

```
[第 3 组 · 词 12/50]  ← 替代原来的 "12/250"
切换到下组: 滑动到底部或点击"下一组"
```

注意：一组完成后才计一次"完成"，学完一组才能解锁下一组（类似不背单词的强制推进）

**工作量：** LearningViewModel + WordBrowseView 约 30 行

#### 2.2 强制复习机制

**新词学习前，必须先完成到期的复习**

```swift
func startSession() {
    // 1. 先取到期复习词（dueRecords）
    // 2. 如果有复习词，sessionWords = dueRecords，不掺新词
    // 3. 复习全完成后才允许学新词
}
```

**工作量：** LearningViewModel.startSession() 约 20 行
**效果：** 不背单词核心机制，避免"只刷不记"

#### 2.3 打卡日历优化

**现有 CalendarView + DailyLog 已有基础，但缺少：**
- 打卡按钮（现在没有主动打卡入口）
- 月度统计（本月学了几天、连续几天）
- DailyLog 的 newWordsLearned/reviewsDone 实际未写入（只在 LearningRecord 记录）

**工作量：** DashboardView + DailyLog 写入约 40 行

#### 2.4 学习目标引导

**选择课程后，引导用户设定每日新词目标，显示预计完成时间**

```
每日 10 词 → CET4 4500 词约 450 天完成
每日 20 词 → CET4 4500 词约 225 天完成
```

**工作量：** SettingsView 约 15 行（已有 dailyGoal）

### Phase 3 — 进阶功能（3-5 天/项）

#### 3.1 真实例句替换

**当前例句是模板生成的（"The team plans to..."），改用 API 拉取真实例句**

方案：
1. 在 expand_wordbanks.py 中增加一个步骤：用 MyMemory API 拉取每个词的例句
2. 或者利用文章阅读中的句子：提取文章中含有目标词的句子作为例句
3. 现有模板例句作为兜底

**工作量：** 脚本修改 + 重新生成词库约半天
**效果：** 不背单词核心体验——真实语境

#### 3.2 单词本

**筛选出 mastery 0/1/2/3 各状态的词，列表展示**

```
全部单词 → 可按课程筛选 → 按掌握状态分组
未学习: 3250 词
学习中: 850 词
已掌握: 320 词
熟练: 80 词
```

点击词条进入 WordDetailView，支持批量操作

**工作量：** 新增 WordListView + ViewModel 约 100 行
**效果：** 红宝书 App 核心功能

#### 3.3 锁屏/小组件

**iOS 桌面 Widget 展示每日一词 + 进度概览**

```swift
// WidgetExtension target + TimelineProvider
// 每日一词：随机从 dueRecords 中取
// 进度概览：今日目标 5/10
```

**工作量：** 新增 Widget Extension target 约 200 行
**效果：** 百词斩锁屏背词类似，碎片时间利用

#### 3.4 词库导入

**支持用户导入自定义词库（JSON/CSV），或输入单词列表自动拉取释义**

**工作量：** 导入界面 + 解析 + 翻译 API 约 150 行
**效果：** 墨墨核心卖点，覆盖纸质书用户

## 四、优先级建议

```
Phase 1（本周）              Phase 2（下周）            Phase 3（后续）
─────────────────────  ─────────────────────  ─────────────────────
✓ 左右滑标记掌握状态         List 分组               真实例句替换
✓ 展示词根词缀              强制复习机制             单词本
✓ 显示掌握状态+复习时间      打卡日历优化             锁屏/小组件
✓ 错题熔断                  学习目标引导             词库导入
```

**建议先做 Phase 1 的四项，每项 1 天内完成，总耗时约 2-3 天。**
这四项都是小改动，但能显著提升用户体验：
- 浏览模式不再只是"看"，可以快速过词（左右滑）
- 每个词都展示词根 + 掌握状态（信息密度提升）
- 错题自动重练（学习深度提升）
